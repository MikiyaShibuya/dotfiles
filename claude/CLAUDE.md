# Global Code Review Checklist

## std::function
- 呼び出し前にnullチェック (`if (func)`) があるか
- 引数は値渡し+ムーブを検討（コピーコストを避ける）
- キャプチャのライフタイムは安全か

## シグネチャ変更時
- 引数の渡し方（値/const参照/右辺値参照）は適切か
- ムーブセマンティクスを活用できるか
- 既存の呼び出し元への影響

## スマートポインタ
- 所有権は明確か
- `shared_ptr` vs `unique_ptr` の選択は適切か
- 循環参照の可能性はないか

## 一般
- 防御的プログラミング: 単体で正しく動作するか（現在の呼び出し元だけでなく）
- エラーハンドリング: 失敗ケースは適切に処理されているか

# Git Safety Rules (CRITICAL)

## 禁止コマンド
以下のコマンドは**絶対に実行禁止**:
- `git reset --hard` - データ消失のリスク
- `git push --force` (main/masterブランチ)
- `git clean -fd` - untrackedファイル消失のリスク

## 確認必須コマンド
以下のコマンドは**未コミットの変更を破棄する可能性がある**ため、実行前にユーザー確認が必要:
- `git restore <file>` (--stagedなし) - ワーキングツリーの変更を破棄
- `git checkout -- <file>` - 同上
- `git stash drop` - stashの内容を削除

**原則**: 未コミットの変更を失う可能性のある操作は、明示的なリストにないものでも、実行前にユーザーに確認する。

## 必須プラクティス
1. **gitコマンド実行前にディレクトリ確認**
   - 必ず `cd /absolute/path && git ...` の形式で明示的に指定
   - サブモジュールと親リポジトリを混同しない

2. **間違ったコミットの修正方法**
   - `git revert` を使用（履歴を保持）
   - または、ユーザーに状況を説明して対処方法を相談

3. **破壊的操作の前はユーザーに確認**
   - reset, force push, rebase, clean など

## コミット作成ルール
1. **適度な粒度でコミットを分割**
   - 一度に大きな変更をコミットせず、論理的な単位で複数のコミットに分ける
   - 各コミットは単一の目的・変更を表すようにする

2. **amendの確認**
   - 最近作られたコミットを微修正する場合、amendするかどうかをユーザーに対話的に確認する
   - 勝手にamendしない

3. **コード・関連ファイルの同期**
   - 機能追加/削除時は、その機能に言及・依存する全てのファイルを同一コミットで更新する
   - 対象: ドキュメント（README, TODO, SPEC, CHANGELOG）、テスト（コード, 仕様書）、
     ビルド設定、CI/CD、サブモジュール
   - 原則: 1つのコミットでcheckoutすれば全てが整合する状態を維持する
   - コミット前チェック: 変更した関数/クラス名でgrepし、更新漏れがないか確認する

4. **untrackedファイルの確認**
   - 不要なuntrackedファイルをコミットしないように注意する

# 過去の再発防止策

## 依存関係の置換時はバージョンを確認する (2026-01-19)
**問題**: ソースビルド（git master）をパッケージ（1.1.0）に置換する提案で、「差分は出ないはず」と断言したが、バージョンが異なることを確認していなかった

**再発防止**:
- インストール方法を変更する際は、**両者のバージョンを必ず確認**してから互換性を判断する
- git masterとaptリポジトリのlatestでのバージョン違いなど、明記されないパッケージバージョンに注意する

## 探索で見つからない場合は範囲を広げるか確認する (2026-01-20)
**問題**: センサー位置の定義場所を調査する際、限定的な検索で見つからなかったにも関わらず「見つからなかった」と結論づけ、`config/robot.py`を見落とした

**再発防止**:
- 何かを探して見つからない場合、以下のいずれかを実施する:
  1. **探索範囲を広げて再検索**: ディレクトリ範囲を広げる、別のキーワードで検索する
  2. **探した範囲と探し方をユーザーに提示**: 「〇〇の範囲で△△を検索しましたが見つかりませんでした。より広く探しますか？」と確認する
- **「見つからなかった」と結論づけない**: 一部を探して見つからなかっただけで「存在しない」と断言しない

## コミット前に変更内容を確認する (2026-01-21)
**問題**: `git status`で表示された全ファイルを確認せずに`git add`し、現在のタスク（センタリング削除）と無関係な変更（以前のセッションで作成したテストファイル）を同一コミットに含めた

**再発防止**:
- **コミット前に各ファイルの変更内容を確認**: `git diff <file>`で変更が現在のタスクに関連するか確認する
- **まとまった内容のみコミット**: 特に指示がない限り、1つのコミットには論理的に関連する変更のみ含める
- **無関係な変更は分離**: 以前のセッションの変更や別タスクの変更が混在している場合、別コミットにするかユーザーに確認する
- **`git add`は慎重に**: 全ファイル一括追加（`git add .`や全modified files）は避け、関連ファイルを選択的に追加する

## ユーザーの曖昧な応答をコミット指示と解釈しない (2026-02-01)
**問題**: 作業報告後にユーザーが「y」と応答したのを、提案していないコミットの承認と解釈し、勝手にコミットを実行した

**再発防止**:
- **コミットはユーザーが明示的に指示した場合のみ実行する**（「コミットして」「commitして」等）
- 「y」「ok」「了解」などの曖昧な応答をコミット指示と解釈しない
- 特に、**自分が提案・質問していないアクションを、ユーザーの応答から勝手に推測して実行しない**

## 「存在する」と「正しい」を混同しない (2026-02-13)
**問題**: rdpwrap.iniに`[10.0.26100.6725]`セクションが「既にある」と判断して追加しなかったが、既存セクションの中身（オフセット値等）はこのマシンのtermsrv.dllに対して不正確だった。ユーザーが正しい内容の同名セクションを追加したら動作した。

**根本原因**: セクションヘッダ（名前/キー）の一致を、内容の正確性と混同した。「存在チェック」だけ行い「正確性チェック」を省略した。

**再発防止**:
- 設定・エントリが「既にある」と判断した場合、**名前/キーの一致だけでなく、内容が目的に合致しているか**を確認する（「存在する」≠「正しい」）
- 動作する状態と動作しない状態の差分があるとき、「意味がないはず」の差分でも、まず事実として内容を比較し列挙する
- 仮定を置いたら明示してユーザーに伝える（例：「6725セクションは既存のもので内容も正しいと判断しましたが、中身を比較しますか？」）
